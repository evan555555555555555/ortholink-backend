0. READ THIS FIRST: Medical Device Industry Primer
This section exists because the engineer building OrthoLink may not know what a 510(k) is or why Ukraine Resolution 753 matters. Read this before touching any code. Every concept referenced later in this document is defined here.
0.1 What Is a Medical Device?
A medical device is any instrument, apparatus, implant, or software intended for diagnosis, prevention, monitoring, treatment, or alleviation of disease or injury. Medical devices range from tongue depressors (Class I, minimal risk) to artificial hip joints (Class III, highest risk). Every country has its own classification system, but they all share the same principle: the higher the risk, the more regulatory evidence required before you can sell it.
0.2 What Is an Orthopedic Device?
Orthopedic devices are implants and instruments used to treat musculoskeletal conditions. BRM Extremities (our first customer) manufactures: joint spacers (temporary implants placed in infected joints), fixation systems (plates, screws, nails that hold fractured bones together), and extremity reconstruction implants (replacement components for fingers, wrists, ankles, toes). These are typically Class IIb or III devices because they are implanted inside the human body, contact bone tissue, and remain for extended periods. This means they face the strictest regulatory requirements in every country.
0.3 The Regulatory Approval Process (Simplified)

0.6 The Distributor Fraud Problem (Why DVA Exists)
When a manufacturer wants to sell in a foreign country, they typically partner with a local distributor who handles registration, customs, and sales. The manufacturer has no way to verify what the distributor tells them because the regulations are in a foreign language, on government websites they cannot navigate, in legal frameworks they do not understand.
Some distributors exploit this by padding the document list — claiming the government requires documents it does not, then charging the manufacturer consulting fees to "prepare" those documents. Others simply send overwhelming lists that cause the manufacturer to abandon the market entirely (which is what happened to BRM in India). This is the core problem DVA solves: cross-reference the distributor's claims against the actual government regulations in the OrthoLink vector store.

2.1 What Went Wrong (Technical Failures)

4. The Five AI Agents
4.1 CrewAI Orchestration Architecture
All five agents are implemented as CrewAI Agent classes (github.com/crewAIInc/crewAI). CrewAI provides: (1) Agent = a role + goal + backstory + tools + LLM config, (2) Task = a specific unit of work with expected_output schema, (3) Crew = a group of agents + tasks executing in a Process (Sequential or Hierarchical), (4) Memory = short-term (within crew), long-term (across sessions), entity (persistent profiles). Each OrthoLink API endpoint instantiates a Crew, kicks it off, and returns the structured Pydantic output.
4.1.1 Why CrewAI Over Raw LLM Calls
Structured delegation: Manager agents assign sub-tasks to specialist agents with explicit expected outputs.
Tool isolation: Each agent has access to only the tools it needs (least privilege).
Memory across sessions: Strategy Agent remembers device profiles. Alert Agent remembers last scan hashes.
Guardrails: CrewAI's output_pydantic forces structured JSON; no free-form hallucination possible.
Observability: CrewAI logs every agent thought, tool call, and delegation decision for audit trails.
Future-proofing: CrewAI Flows (pipeline orchestration) enables chaining crews into multi-step workflows without custom code.

4.2 Agent 1: DVA — Distributor Verification Agent ("The Sentinel")
CrewAI Config: role='Distributor Verification Specialist' | goal='Detect fraudulent document requests' | process=Sequential
4.2.1 Processing Pipeline (7 Steps)

Step | Operation | Implementation | Output
1. Parse | Extract document names from uploaded CSV | pandas.read_csv(); normalize: lowercase, strip whitespace, expand abbreviations ("Cert"→"Certificate") | List[str]: normalized document names
2. Retrieve Baseline | Get all regulatory requirements for country+class from vector store | FAISS search_vector_store(country=X, device_class=Y, k=50). Metadata filter: is_active=true | List[RegRequirement]: the ground-truth set
3. Embed Items | Generate embeddings for each distributor item | OpenAI text-embedding-3-large. Batch API call. Cache embeddings for repeat items. | ndarray[n, 3072]
4. Semantic Match | For each distributor item, find closest baseline item | Cosine similarity. Threshold: ≥0.82 = candidate match. <0.82 = no match. | List[Tuple[dist_item, baseline_item, score]]
5. LLM Classify | Confirm/reject match, assign status, generate citation | gpt-4o structured output (JSON mode). Prompt includes regulation text. Output: {status, citation, confidence, explanation} | List[Classification]
6. Gap Detect | Find baseline items with no distributor match | Set difference: baseline - matched_baseline = MISSING items | List[MissingItem]
7. Score | Compute fraud risk and flag low-confidence items | fraud_score = (EXTRA + MISSING) / total. Items with confidence < 0.7 → needs_human_review queue | GapAnalysisReport (Pydantic)

4.2.2 Tools (Python @tool Functions)

Tool Name | Signature | What It Does
search_vector_store | (country, device_class, k=50) → List[Dict] | FAISS query with metadata filtering. Returns chunks with text, citation, section_path.
compute_similarity | (item_embedding, baseline_embeddings) → List[Tuple[str, float]] | Cosine similarity via numpy. Returns sorted pairs above threshold.
classify_status | (distributor_item, matched_regulation, regulation_text) → Dict | gpt-4o JSON mode. Returns {status, citation, confidence, explanation}.
calculate_fraud_score | (classifications) → float | Aggregate: (EXTRA + MISSING) / total. >0.3 triggers warning.

4.3 Agent 2: CRA — Compliance Review Agent ("The Reviewer")
CrewAI Config: role='Medical Device Compliance Auditor' | goal='Stream compliance reviews grounded in regulatory text' | process=Sequential
Built on: github.com/Dineshkumar-Ponnusamy/Compliance-Reviewer (React 18 + TypeScript + Vite). OrthoLink adds the FastAPI backend, RAG grounding, and resolves all 22 gaps from the build brief (see Section 7).
4.3.1 How It Works
User uploads a document (PDF, DOCX, XLSX, TXT) via the dashboard.
Frontend sends file to POST /api/v1/review-document with selected standard(s) and country.
FastAPI extracts text (via Web Worker on frontend OR server-side fallback).
CRA agent chunks the document, retrieves relevant regulatory sections from vector store (RAG grounding).
Agent reviews document against regulatory requirements via gpt-4o with streaming (SSE).
Findings stream to frontend in real-time: AIComment[] with severity, line numbers, clause violated.
AIRecommendation[] generated with actionable remediation steps.
Revised text shown in side-by-side diff viewer (react-diff-viewer-continued).
Everything persisted to audit_log (append-only, tamper-proof).

4.6 Agent 5: RAA — Regulatory Alert Agent ("The Watchman")
CrewAI Config: role='Regulatory Change Intelligence Analyst' | goal='Detect regulation changes, archive old versions, alert affected users' | process=Sequential
Adapted from the ai-news-agent pattern (source monitoring → filter → prioritize → summarize). Instead of Bing News, monitors government gazette sites and regulatory agency databases.
4.6.1 Pipeline: Scrape → Diff → Classify → Archive → Re-Embed → Notify
Scrape: Re-fetch regulation text from government source URL (BeautifulSoup; Selenium fallback for JS sites).
Diff: SHA-256 hash comparison of cleaned text against stored chunk_hash. If hash changed, identify specific changed sections.
Classify: gpt-4o assesses change severity (Critical/Major/Minor) and identifies affected device classes.
Archive: Mark old chunks is_active=false, set valid_to=today. NEVER delete old versions. (Audit requirement.)
Re-Embed: Ingest changed sections through full pipeline (clean, chunk, translate, embed). New chunks get is_active=true.
Notify: Send email via Resend + in-app notification to users subscribed to that country/class combination.
4.6.2 Monitoring Schedule
Daily: United States (FDA Federal Register, eCFR), European Union (EUR-Lex, Official Journal)
Every 3 days: UK, Canada, Japan, Australia, India, Brazil
Weekly: Ukraine, China (manual check), Russia (manual check), Switzerland, Mexico, South Korea, Saudi Arabia
4.6.3 Innovation: CrewAI Memory for Change Context
The Alert Agent uses CrewAI's long-term memory to remember previous changes. When a new change is detected, the agent can reference past changes to assess cumulative impact. For example: "This is the third amendment to EU MDR Annex IX in 6 months. Cumulative impact on Class III device manufacturers is now Critical." This contextual alerting is something no competitor offers.

9.2 FAISS Vector Store: Chunk Metadata Schema
Every embedded chunk carries structured metadata for filtering and citation. This metadata is what makes OrthoLink's retrieval precise rather than generic.

Field | Type | Example | Used By
country | string | Ukraine | All agents: country-level filtering
document_id | string | UA-RES-753 | Citation generation
document_title | string | Resolution 753: Technical Regulation on Medical Devices | Human-readable source
section_path | string | Part II > Chapter 3 > Article 12 > Clause 12.3 | Full hierarchy for precise citation
article_number | string | 12 | Article-level filtering
clause_number | string | 12.3.1 | Clause-level precision
device_classes_applicable | list[str] | ["IIa", "IIb", "III"] | Filter by device class
valid_from | date | 2024-01-15 | Version management
valid_to | date|null | null (active) | null = current version
is_active | boolean | true | Only active chunks returned by default
language_original | string | uk | ISO 639-1 code
chunk_hash | string | sha256:abc123... | Change detection by Alert Agent
parent_chunk_id | string|null | UA-RES-753-ART-12 | Parent Document Retriever linkage

9.3 Document Ingestion Pipeline (Per Country)
Acquire: Scrape government website (BeautifulSoup; Selenium fallback for JS-rendered). For geo-blocked sources (China NMPA, Russia Roszdravnadzor): manual PDF upload.
Validate: Scraper validation step. Minimum word count (>500 words). Presence of legal keywords ("Article", "requirement", "manufacturer"). Log WARNING if sanity check fails. Do not embed garbage.
Clean: Strip navigation, cookie banners, browser warnings, footer boilerplate. If clean_text contains "Firefox" or "Google Chrome", the scraper has failed. (Previous contractor's scraper did not catch this.)
Translate: Non-English docs → English via gpt-4o with regulatory term preservation. Original language retained in metadata. NOT Google Translate (loses legal precision).
Chunk: Split at Article/Section boundaries (NOT token count). Every chunk carries full metadata schema. Parent Document Retriever pattern: child clauses for search, full parent article for generation.
Embed: text-embedding-3-large (3072 dims) → FAISS with metadata index. Persist index to disk.
Audit: After ingestion, run chunk_audit.py: sample 10 random chunks, verify metadata correctness, verify text is clean legal content. Log results. This is how we caught the previous contractor's failures.
